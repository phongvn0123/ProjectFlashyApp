<!-- 01-RESEARCH.md § "Firestore Schema: 5 Root Collections" (~line 469-508) defines the 5 root collections. Per ROADMAP's explicit "5 root collections" success criterion, quiz attempts are kept as a SUBCOLLECTION `quizzes/{quizId}/attempts/{attemptId}` — do NOT promote quiz_attempts to a 6th root collection, even though 01-RESEARCH.md's later "Alternative Architecture for Quiz Attempts" aside suggests it; that aside would violate the explicit 5-collection requirement and is not used here. -->

The 5 root collections and their subcollections (exact names):
1. `users/{uid}` — no subcollections
2. `flashcard_sets/{setId}` — subcollection `flashcards/{cardId}`
3. `classrooms/{classId}` — subcollections `members/{userId}`, `assigned_sets/{setId}`, `activities/{activityId}`
4. `quizzes/{quizId}` — subcollections `questions/{questionId}`, `attempts/{attemptId}`
5. `user_sessions/{sessionId}` — subcollection `cards/{cardSessionId}`

<!-- 01-RESEARCH.md's Common Pitfalls § "Pitfall 5: Firestore Security Rules Deny All in Production" (~line 1460-1567) has a rules TEMPLATE that must NOT be copied verbatim for the `users/{userId}` block — its `allow write: if isAdmin();` would block a signed-in user from editing their OWN profile (name/phone/address/avatar), which contradicts Phase 2's PROF-02 requirement. Use the corrected rule below instead. -->

Exact `users/{userId}` rule to write (corrected from the template, allows self-edit but locks role/status to admin-only):
- `allow read: if isSignedIn() && (request.auth.uid == userId || isAdmin());`
- `allow create: if isSignedIn() && request.auth.uid == userId;`
- `allow update: if isSignedIn() && ((request.auth.uid == userId && request.resource.data.role == resource.data.role && request.resource.data.status == resource.data.status) || isAdmin());`
- `allow delete: if isAdmin();`

`flashcard_sets`/`classrooms`/`user_sessions` rules: use 01-RESEARCH.md's Pitfall 5 template as-is (owner/public-read, owner-write for sets; teacher-owns/member-reads for classrooms; own-only for user_sessions) — those blocks are correct and do not need modification.

`quizzes/{quizId}` + subcollections rule (not in the Pitfall 5 template — write new, matching the chosen subcollection-attempts architecture):
- `allow read: if isSignedIn() && (resource.data.teacher_id == request.auth.uid || resource.data.status == 'published');`
- `allow create, update, delete: if isSignedIn() && isTeacher() && resource.data.teacher_id == request.auth.uid;` (for `create`, since `resource` doesn't exist yet, use `request.resource.data.teacher_id == request.auth.uid` instead)
- `match /questions/{questionId} { allow read: if isSignedIn(); allow write: if isTeacher(); }`
- `match /attempts/{attemptId} { allow read: if isSignedIn() && resource.data.user_id == request.auth.uid; allow create: if isSignedIn() && request.resource.data.user_id == request.auth.uid && request.resource.data.status == 'in_progress'; allow update: if isSignedIn() && resource.data.user_id == request.auth.uid && resource.data.status == 'in_progress'; }` (the `status == 'in_progress'` update guard is what will let Phase 6 enforce QUZ-15 "cannot retake a submitted quiz" at the rules layer, not just in the UI)

4 composite indexes for `firestore.indexes.json`:
1. `flashcard_sets`: `owner_id ASC, created_at DESC`
2. `classrooms`: `teacher_id ASC, created_at DESC`
3. `user_sessions`: `user_id ASC, started_at DESC`
4. Collection group `attempts` (queryScope `COLLECTION_GROUP`, since attempts is a subcollection under multiple quizzes): `user_id ASC, started_at DESC` — supports a student's "all my quiz attempts across every quiz" list view (Phase 6)

| Boundary | Description |
|----------|--------------|
| App ↔ Firestore (emulator or real) | All reads/writes cross this boundary; rules are the only enforcement point since the client SDK is fully trusted-input-capable (any authenticated user can attempt any request) |
| Emulator ↔ real project | `.firebaserc` pins to `demo-memocard-dev` (fake); real project credentials never touch this plan's automated work |
