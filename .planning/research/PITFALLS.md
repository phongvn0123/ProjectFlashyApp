## Critical Pitfalls

### Pitfall 1: SQLite FFI Initialization Forgotten on Windows

**What goes wrong:**  
App runs fine on Android emulator, launches on Windows desktop, crashes immediately with cryptic FFI error: `"Invalid argument(s): Failed to load dynamic library 'sqlite3.dll'"` or `"SqfliteFfiException(error)"`. The database layer is completely inaccessible.

**Why it happens:**  
`sqflite_common_ffi` requires **two** initialization steps that are easy to do partially:
1. Call `sqfliteFfiInit()` (initializes the C-level sqlite3 loader)
2. Set `databaseFactory = databaseFactoryFfi;` (tells sqflite to use FFI instead of platform channel)

Developers often forget step 1 and just set `databaseFactory`, which fails silently on Android but crashes on Windows when the FFI layer isn't initialized.

**How to avoid:**  
In `main.dart`, **before** `runApp()`:
```dart
import 'dart:io';
import 'package:sqflite_common_ffi/sqflite_ffi.dart';

void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  
  // CRITICAL: Windows + Linux need explicit FFI init
  if (Platform.isWindows || Platform.isLinux) {
    sqfliteFfiInit();
  }
  
  // Set factory AFTER init (order matters)
  databaseFactory = databaseFactoryFfi;
  
  // Then initialize other services
  await Firebase.initializeApp(options: DefaultFirebaseOptions.currentPlatform);
  
  runApp(const MyApp());
}
```

**Code checkpoint:** In Phase 1 (Foundation), add a Windows-specific test that opens a database and writes a test row. If this fails, you'll know FFI isn't wired.

**Warning signs:**  
- App works on Android, crashes on Windows with database errors
- Error message mentions `Failed to load dynamic library` or `'sqlite3.dll'`
- Windows build completes but fails at runtime (not a build error)
- Only Windows developers encounter the issue (Android devs think it's platform-specific)

**Phase to address:** Phase 1 (Foundation) — before any developer writes a single repository method. Verify on both Android and Windows in the same commit.

### Pitfall 2: Firebase on Windows Desktop Crashes at Startup

**What goes wrong:**  
Windows build compiles successfully. On launch, app crashes with:
```
[core/not-initialized] Firebase has not been correctly initialized.
```
or
```
CMake error / linking error during build
```

This is unrelated to sqflite. The crash happens because `firebase_core` initialization doesn't handle Windows the way it handles Android or web.

**Why it happens:**  
- `firebase_core` on Windows requires **explicit platform options** (API key, project ID, auth domain) passed to `Firebase.initializeApp()`, similar to web initialization. 
- Android and iOS use platform channels to read options from native config. Windows has no platform channel for Firebase config.
- Developers copy an Android-only initialization and don't branch for Windows.
- Some Firebase packages (e.g., `firebase_messaging`) have **no Windows support at all**, causing build failures if not handled.

**How to avoid:**  
Use `flutterfire configure` **with Windows support enabled**:
```bash
flutterfire configure --platforms=android,ios,web,windows
```

In `firebase_options.dart` (auto-generated by `flutterfire configure`), verify Windows options are present:
```dart
static const FirebaseOptions windows = FirebaseOptions(
  apiKey: 'YOUR_API_KEY',
  appId: 'YOUR_APP_ID',
  messagingSenderId: 'YOUR_SENDER_ID',
  projectId: 'YOUR_PROJECT_ID',
  authDomain: 'YOUR_PROJECT.firebaseapp.com',
);
```

In `main.dart`, branch explicitly:
```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  
  if (Platform.isWindows || Platform.isLinux) {
    sqfliteFfiInit();
  }
  databaseFactory = databaseFactoryFfi;
  
  // Firebase.initializeApp MUST use platform-specific options on Windows
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );
  
  runApp(const MyApp());
}
```

**Do NOT add `firebase_messaging`** to this project — it is incompatible with Windows desktop. Project scope already excludes FCM.

**Warning signs:**  
- App runs on Android, crashes on Windows with "Firebase not initialized"
- Firebase build fails with CMake errors on Windows
- Using `firebase_messaging` anywhere in the codebase
- Error message includes "linking error" or "unresolved external symbol" during Windows build

**Phase to address:** Phase 1 (Foundation) — immediately after sqflite setup. Test Firebase initialization on Windows before any authentication flow is built.

**Recovery:** Delete `firebase_messaging` from pubspec.yaml. Re-run `flutterfire configure --platforms=android,ios,web,windows`. Test Windows launch before committing.

### Pitfall 3: Riverpod autoDispose Confusion — Data Disappears When Unwatched

**What goes wrong:**  
Data loads correctly when a screen is open and being watched. User navigates away. The Riverpod provider holding that data is disposed. User navigates back. The exact same data must load again from Firestore instead of being cached. This creates unnecessary reads and slow UX.

Alternatively: A provider is watched in one place, not watched elsewhere. The developer assumes the data persists but it's been disposed, causing stale data bugs.

**Why it happens:**  
Riverpod 2.5+ uses `autoDispose` by default in code-generated providers (`@riverpod`). This is sensible for memory, but creates confusion:
- Old code without `autoDispose` explicitly keeps state forever
- New code-generated providers throw away state immediately
- When a provider has zero listeners for one frame, it disposes automatically
- Developers don't know when this happens and assume state persists

**How to avoid:**  
**Use explicit `@Riverpod(keepAlive: true)` for data that should persist** (e.g., user profile, flashcard sets):

```dart
@riverpod(keepAlive: true)
Future<User> currentUser(CurrentUserRef ref) async {
  return await _fetchUser();
}

@riverpod  // Auto-disposes: fine for temporary UI state
Future<List<FlashcardSet>> userFlashcardSets(UserFlashcardSetsRef ref) async {
  final user = ref.watch(currentUserProvider);
  return user.when(
    data: (u) => _fetchFlashcardSets(u.id),
    error: (e, st) => throw e,
    loading: () => throw Exception('User not loaded'),
  );
}
```

**Document the rule:** "If data comes from an expensive remote fetch or must persist across screens, use `keepAlive: true`. If it's temporary state (loading indicator, form input), use default `autoDispose`."

**For StreamProvider with Firestore:** Always use `.autoDispose` and proper cleanup:
```dart
@riverpod
Stream<List<Flashcard>> flashcardStream(FlashcardStreamRef ref) async* {
  final setId = ref.watch(selectedSetIdProvider);
  
  if (setId == null) yield [];
  else {
    final subscription = FirebaseFirestore.instance
        .collection('flashcard_sets').doc(setId)
        .collection('flashcards')
        .snapshots();
    
    ref.onDispose(() => subscription.listen((_) {}).cancel());
    
    yield* subscription.map((snap) => 
        snap.docs.map((d) => Flashcard.fromFirestore(d)).toList());
  }
}
```

**Warning signs:**  
- Screens show stale data after navigating back
- Data loads twice on one screen (initial load + unnecessary reload)
- Developers complain: "The data was there but now it's gone"
- Firestore read count is higher than expected for the app's actual usage

**Phase to address:** Phase 1 (Foundation) — establish the `keepAlive` rule in the provider template before any feature work. Code review check: "Is this `autoDispose` by accident?"

### Pitfall 4: Firestore N+1 Reads Explode Spark Tier Quota

**What goes wrong:**  
The app works perfectly in development with 2–3 test users. During testing with 5 developers + demo day, **Firestore starts rejecting writes** with quota exceeded errors. The app becomes completely non-functional for demo. Investigation reveals 50K reads consumed in a few hours, when the Spark tier limit is exactly **50K reads per day**.

**Why it happens:**  
Classic N+1 pattern in Firestore queries:
1. Fetch all `FlashcardSet` documents for a user: 1 read
2. For each flashcard set, fetch all `Flashcard` children: N reads
3. For each flashcard, fetch `CardProgress` for this user: N reads

A single "load my flashcard library" can trigger 100+ reads.

Combined with 5 developers simultaneously testing + the Riverpod listener leaks mentioned below, quota exhaustion happens in hours, not days.

**How to avoid:**  

**Strategy 1: Denormalize data into parent document** (for small sets)
```dart
// Instead of: FlashcardSet has subcollection Flashcard
// Do this: Store flashcards array inside FlashcardSet if <10 items

class FlashcardSet {
  final String id;
  final String name;
  final List<Flashcard> flashcards; // Embed instead of subcollection
}
```

**Strategy 2: Use collection group queries ONLY with strict filters**
```dart
// WRONG: Fetches every flashcard in the DB
final allFlashcards = FirebaseFirestore.instance
    .collectionGroup('flashcards')
    .snapshots();

// CORRECT: Fetch flashcards for one set only
final setFlashcards = FirebaseFirestore.instance
    .collection('flashcard_sets')
    .doc(setId)
    .collection('flashcards')
    .snapshots();
```

**Strategy 3: Batch fetch with document references** (one read, not N)
```dart
// Instead of: fetch each Classroom -> fetch each AssignedSet separately
class Classroom {
  final String id;
  final List<DocumentReference> assignedSetRefs; // Store refs, not docs
}

// Fetch: 1 read for classroom + 1 batch get for all assigned sets
final classroom = await _db.collection('classrooms').doc(id).get();
final sets = await _db.getAll(classroom.data()['assignedSetRefs']);
```

**In data layer, add read cost documentation:**
```dart
/// Fetch user's flashcard library.
/// Firestore cost: 1 read (sets) + N reads (flashcards per set) + N*M reads (progress).
/// ~100 reads for 10 sets with 20 cards each + progress tracking.
/// Mitigation: Denormalize CardProgress into each Flashcard subdoc once per session.
Future<List<FlashcardSetWithCards>> getUserFlashcardLibrary(String userId) async {
  // ...
}
```

**Phase 1 & 2:** Establish a Firestore schema that avoids N+1 by design. Require code review to document read cost for any multi-level query.

**Warning signs:**  
- Firestore reads spike to 40K+ per day during normal testing
- Read count per feature is unknown (developers don't measure it)
- Multiple developers testing simultaneously causes quota errors
- Demo day is on a day when quota reset hasn't happened yet (midnight Pacific)

**Phase to address:** Phase 1 (Foundation) — before any repository methods are written. Establish data fetching patterns that minimize reads.

### Pitfall 5: Offline-First Cache Becomes Stale and Corrupts User Trust

**What goes wrong:**  
User opens flashcard set offline, sees outdated content (teacher updated it yesterday). User goes online, refreshes, sees the new content. But if cache invalidation is not explicit, user might keep seeing stale data on subsequent offline loads. Or: User's offline edits sync to Firestore, but local cache isn't updated, creating a divergence where the app shows different data than what's actually stored.

**Why it happens:**  
The project uses an explicit offline-first pattern: **Firestore is source of truth, SQLite is cache**. This is correct, but requires **disciplined cache invalidation**:
- When data syncs from Firestore to SQLite, the timestamp must be updated
- When user makes changes, they must be written to Firestore AND the local cache must be refreshed
- If Firestore sync fails, the app must clearly indicate "cached data may be stale"

Developers often forget to refresh the cache after a write, or they rely on Riverpod `autoDispose` to throw away old data, expecting it to reload fresh — but if there's no network, it loads the stale SQLite.

**How to avoid:**  

**Establish a cache invalidation rule in the repository layer:**
```dart
class FlashcardRepository {
  final FirebaseFirestore _firestore;
  final Database _local;
  final Ref _riverpod;
  
  /// Write flashcard locally and to Firestore. Always sync.
  Future<void> createFlashcard(Flashcard card) async {
    // 1. Write to local DB first (for instant UX)
    await _local.insert('flashcards', card.toMap());
    
    // 2. Write to Firestore
    try {
      await _firestore
          .collection('flashcard_sets').doc(card.setId)
          .collection('flashcards').doc(card.id)
          .set(card.toFirestore());
      
      // 3. Mark sync status as synced in local DB
      await _local.update(
        'flashcards',
        {'is_synced': 1, 'synced_at': DateTime.now().toIso8601String()},
        where: 'id = ?',
        whereArgs: [card.id],
      );
      
      // 4. Invalidate Riverpod cache so next watch fetches fresh
      _riverpod.invalidate(flashcardStreamProvider(card.setId));
    } catch (e) {
      // Firestore failed, mark as unsynced
      await _local.update(
        'flashcards',
        {'is_synced': 0, 'error': e.toString()},
        where: 'id = ?',
        whereArgs: [card.id],
      );
      rethrow;
    }
  }
}
```

**In the UI, show clear offline status:**
```dart
Consumer(builder: (context, ref, child) {
  final flashcards = ref.watch(flashcardStreamProvider(setId));
  
  return flashcards.when(
    data: (cards) {
      final unsynced = cards.where((c) => c.isSynced == false).toList();
      return Column(
        children: [
          if (unsynced.isNotEmpty)
            Banner(
              message: 'Pending: ${unsynced.length} unsynced changes',
              location: BannerLocation.topEnd,
              child: FlashcardList(cards),
            )
          else
            FlashcardList(cards),
        ],
      );
    },
    error: (e, st) => Text('Error: $e'),
    loading: () => LoadingSpinner(),
  );
});
```

**Phase 2:** Define the `is_synced` / `synced_at` schema in SQLite before any write operations are built. Add a sync status indicator to relevant UI screens.

**Warning signs:**  
- Users report stale data after editing offline
- No visibility into which records are unsynced
- Cache invalidation is inconsistent across the app (some places invalidate, some don't)
- Developers manually refresh to fix data divergence

**Phase to address:** Phase 1 (Foundation) — before the first flashcard write is implemented.

### Pitfall 6: Firestore Listener Leaks via StreamProvider Without autoDispose

**What goes wrong:**  
App launches, loads a flashcard set, opens a StreamProvider listening to Firestore changes. User navigates to another screen. The listener keeps running in the background, consuming memory and network. If this happens 10 times (10 screens visited, 10 listeners started), memory grows and the app becomes sluggish or crashes. Plus: Firestore shows 100+ simultaneous open connections, even though the user is idle.

**Why it happens:**  
A `StreamProvider` that wraps `FirebaseFirestore.snapshots()` will keep listening **forever** unless explicitly disposed. Without `.autoDispose` modifier, the stream subscription never cancels. Riverpod's default behavior (before 2.5) kept all providers alive, and even in 2.5+, custom StreamProviders can accidentally keep listeners alive.

**How to avoid:**  

**Always use `.autoDispose` on Firestore StreamProviders:**
```dart
// WRONG: Listener never closes
@riverpod
Stream<List<Flashcard>> flashcards(FlashcardsRef ref) async* {
  yield* FirebaseFirestore.instance
      .collection('flashcard_sets').doc(setId)
      .collection('flashcards')
      .snapshots()
      .map((snap) => snap.docs.map(Flashcard.fromDoc).toList());
}

// CORRECT: Listener closes when no one watches
@riverpod
Stream<List<Flashcard>> flashcards(FlashcardsRef ref) async* {
  yield* FirebaseFirestore.instance
      .collection('flashcard_sets').doc(setId)
      .collection('flashcards')
      .snapshots()
      .map((snap) => snap.docs.map(Flashcard.fromDoc).toList());
}
```

Actually, both look the same above — the key is in **usage**. The second one should be auto-disposing by default if generated with `@riverpod`. But if using manual providers, use `.autoDispose`:

```dart
final flashcardsProvider = StreamProvider.autoDispose<List<Flashcard>>((ref) {
  return FirebaseFirestore.instance
      .collection('flashcard_sets').doc(setId)
      .collection('flashcards')
      .snapshots()
      .map((snap) => snap.docs.map(Flashcard.fromDoc).toList());
});
```

**In manual provider setup, add `ref.onDispose` for cleanup:**
```dart
final flashcardsProvider = FutureProvider.autoDispose<List<Flashcard>>((ref) async {
  final subscription = FirebaseFirestore.instance
      .collection('flashcard_sets').doc(setId)
      .collection('flashcards')
      .snapshots()
      .listen((snap) {
        // process
      });
  
  ref.onDispose(subscription.cancel);
  
  // Return current data or empty list
  return [];
});
```

**Phase 1:** Establish the rule: "All Firestore listeners must use `.autoDispose` or manual `ref.onDispose()` cleanup." Code review every StreamProvider.

**Warning signs:**  
- App memory usage grows over time during navigation
- Firestore console shows dozens of simultaneous active listeners, even when app is idle
- App crashes with low memory warnings after visiting many screens
- Switching between screens causes visible lag

**Phase to address:** Phase 1 (Foundation) — before Firestore providers are created.

### Pitfall 7: Firestore Composite Index Missing at Demo Time

**What goes wrong:**  
The app works fine in development. During demo (or after deploy to test environment), a query fails with:
```
FAILED_PRECONDITION: The query requires an index. You can build it using the command below.
<link to build index>
```

The link goes to the Firebase console, but building an index takes 3–5 minutes. If the demo is live and the grader is watching, the app is broken.

**Why it happens:**  
Firestore requires composite indexes (indexes on multiple fields) for queries that filter AND sort by multiple fields simultaneously. The development environment auto-creates these indexes when you hit the query, but **they're not synced to production/test environments by default**. Each Firebase project gets its own set of indexes.

If one developer builds the index locally and another checks out the code without running the same query, the second developer's Firebase project (or the grader's test project) doesn't have the index. The query fails.

**How to avoid:**  

**1. Export and commit Firestore indexes to git:**
In Firebase console: Settings → Indexes → Download as JSON. Save to `.planning/firebase/indexes.json`. Commit to git.

Before demo day, deploy indexes to the test/grading Firebase project:
```bash
firebase deploy --only firestore:indexes
```

**2. Document all compound queries with their required indexes:**
```dart
/// Fetch quiz assignments for a student, sorted by created date (newest first).
/// Requires composite index: (classroom_id, student_id, created_at DESC)
/// TODO: Verify index exists in test project before demo
Future<List<QuizAssignment>> getStudentQuizzes(String classId, String studentId) async {
  return FirebaseFirestore.instance
      .collection('quiz_assignments')
      .where('classroom_id', isEqualTo: classId)
      .where('student_id', isEqualTo: studentId)
      .orderBy('created_at', descending: true)
      .snapshots()
      .first
      .then((snap) => snap.docs.map(QuizAssignment.fromDoc).toList());
}
```

**Phase 2:** Before writing the first compound query, establish the rule: "Every `where() + orderBy()` query must be tested on both local and test Firestore projects. If index is missing, add it to `.planning/firebase/indexes.json` and deploy."

**Warning signs:**  
- Query works on dev machine, fails on another machine with same code
- Grader's demo machine hits a "missing index" error
- Developers have different sets of indexes in their local projects
- No documentation of which queries need which indexes

**Phase to address:** Phase 2 (API & Data) — before the first compound query is implemented.

### Pitfall 8: 5 Developers, 1 pubspec.yaml, Merge Hell

**What goes wrong:**  
Developer A adds `uuid: ^1.0.0` to pubspec.yaml and commits. Developer B adds `intl: ^0.18.0` in parallel. When B pulls and merges, git shows a conflict in pubspec.yaml. Manual merge is error-prone (easy to delete a line accidentally). Now the app is missing a dependency.

Or: Two developers both try to add the same package with different versions (`uuid: ^1.0.0` vs `uuid: ^1.1.0`). After merging, the version constraint is ambiguous or wrong.

**Why it happens:**  
pubspec.yaml is a single file that all 5 developers modify. Without discipline, concurrent modifications cause conflicts. Dart's dependency resolution is strict — even a typo breaks `flutter pub get`.

**How to avoid:**  

**1. Only one person adds dependencies to pubspec.yaml per day.** Establish a rule: "If you need a new package, ask in the group chat. One person (e.g., person-of-the-day) adds it to pubspec.yaml, commits, and pushes. Others pull before adding theirs."

**2. Use exact versions for critical packages, caret (^) for flexible ones:**
```yaml
dependencies:
  flutter:
    sdk: flutter
  firebase_core: ^4.8.0         # Caret: accept 4.x but not 5.x
  cloud_firestore: 4.13.0       # Exact: no surprises mid-project
  riverpod: 2.5.0
  freezed_annotation: ^2.4.0
  
dev_dependencies:
  build_runner: ^2.4.0
  freezed: ^2.4.0
  riverpod_generator: ^2.5.0
```

**3. Designate one "dependency manager" person per sprint.** Only they modify pubspec.yaml. Others request via issue or chat.

**4. Use `.gitattributes` to reduce noise:**
```
pubspec.lock merge=union
pubspec.yaml merge=union
```

This tells git to accept both versions in a conflict rather than ask, but **you must verify the result manually** (run `flutter pub get` to validate).

**Phase 1:** Before any code is written, establish the pubspec.yaml rule. Create a CONTRIBUTING.md document:
```markdown
## Adding Dependencies

1. Check if the package already exists in pubspec.yaml
2. Ask @dependency-manager to add it (one person per day)
3. Wait for them to commit and push
4. Pull and verify `flutter pub get` succeeds
5. Only then start using the package
```

**Warning signs:**  
- pubspec.yaml has merge conflicts every day
- Dependencies mysteriously disappear after a merge
- `flutter pub get` fails with "package not found" errors
- Multiple people adding packages to pubspec.yaml simultaneously

**Phase to address:** Phase 1 (Foundation) — establish the rule before the first dependency is added.

### Pitfall 9: Generated Files (*.g.dart, *.freezed.dart) Committed vs. Gitignored Causes Conflicts

**What goes wrong:**  
Developer A commits `model.freezed.dart` (generated file). Developer B modifies `model.dart` and regenerates locally, creating a different `model.freezed.dart`. When B tries to push, git shows a conflict in the generated file. Developer B has two choices:
1. Use their generated file (but it doesn't match A's code changes)
2. Use A's generated file (but it doesn't match their code changes)

Either way, the generated file is now out of sync with the source. Build fails with "type mismatch" errors.

**Why it happens:**  
Generated files are reproducible — `flutter pub run build_runner build` should always generate the same file from the same source. But git doesn't know this. If the file is committed, git treats it like source code and marks any changes as conflicting. If the file is ignored, one developer might forget to regenerate after a pull, and code fails to compile.

**How to avoid:**  

**1. IGNORE all generated files. Always regenerate locally after `git pull`.**

In `.gitignore`:
```
*.g.dart
*.freezed.dart
*.config.dart
```

In `.gitattributes` (for the rare case where generated files slip in):
```
*.g.dart linguist-generated=true
*.freezed.dart linguist-generated=true
```

**2. Add a pre-commit hook (or CI step) to prevent accidental commits:**
In `.git/hooks/pre-commit`:
```bash
#!/bin/bash
if git diff --cached | grep -E '\.(g|freezed)\.dart'; then
  echo "ERROR: Generated files (.g.dart, .freezed.dart) should not be committed."
  echo "Run: flutter pub run build_runner build"
  echo "Then: git add *.dart (only source files)"
  exit 1
fi
```

**3. Add to CONTRIBUTING.md:**
```markdown
## After git pull

Always regenerate generated files:
```bash
git pull
flutter pub get
flutter pub run build_runner build
```

Do not commit `*.g.dart` or `*.freezed.dart` files.
```

**Phase 1:** Before any models are created with `@freezed`, establish the rule and add the hook.

**Warning signs:**  
- `git status` shows changes to `.g.dart` or `.freezed.dart` files after every `git pull`
- Merge conflicts in generated files are common
- Developers complain: "Type doesn't match after I merged"
- Build fails with "type X is not a subtype of Y" after pulling

**Phase to address:** Phase 1 (Foundation) — establish the rule before any `@freezed` model is created.

### Pitfall 10: Firestore Security Rules Not Written, Anyone Can Read/Write

**What goes wrong:**  
During demo, the grader notices they can access other students' quiz results via the web console. Or: A student figures out they can read other classrooms' flashcards by guessing the document ID. The app is a security failure.

Alternatively: The app launches, and the developer leaves the default Firestore rules (allow read/write for all) in place, forgetting to enable authentication rules before deployment. By demo time, the database is completely open.

**Why it happens:**  
Security rules are boring and blocking features are fun. Developers prioritize building features and defer security rules. By the time they think about rules, there's 1 day to demo and 100 things to test. Rules don't get the attention they need.

**How to avoid:**  

**1. Write basic security rules in Phase 1, before any data is written to Firestore:**
```javascript
// firestore.rules (checked into git)
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // User can only read/write their own document
    match /users/{userId} {
      allow read, write: if request.auth.uid == userId;
    }
    
    // User can read their own flashcard sets
    match /flashcard_sets/{setId} {
      allow read: if resource.data.owner_id == request.auth.uid;
      allow write: if request.auth.uid != null && request.auth.uid == resource.data.owner_id;
      
      // Flashcards in a set: inherit parent read/write
      match /flashcards/{flashcardId} {
        allow read: if request.auth.uid != null;
        allow write: if get(/databases/$(database)/documents/flashcard_sets/$(setId)).data.owner_id == request.auth.uid;
      }
    }
    
    // Add other collections...
  }
}
```

**2. Test rules before writing any data:**
Use Firebase Console → Rules Playground:
- Simulate read as user1 → should succeed only for user1's docs
- Simulate read as user2 → should fail
- Simulate write as anonymous → should fail

**3. Add to the Definition of Done (before any feature is merged):**
- [ ] If this feature reads/writes Firestore, are the security rules updated?
- [ ] Have rules been tested in the Rules Playground?
- [ ] Grader cannot access other users' data

**Phase 2:** Write complete security rules before the first user is created in Firestore. Review with entire team.

**Warning signs:**  
- Firestore rules still say `allow read, write: if request.time < timestamp.value('2025-01-01T00:00:00Z')`
- No authentication check (`request.auth == null` is allowed anywhere)
- Developers haven't tested rules at all
- Rules are written 1 day before demo

**Phase to address:** Phase 2 (API & Data) — immediately after Firestore schema is finalized, before any data is written.

### Pitfall 11: 5 Devs on One Firebase Project, Quotas Collide and Names Conflict

**What goes wrong:**  
Developer A is testing user registration. Developer B is testing quiz grading. Developer C is testing flashcard import. All are hitting the same Firestore Spark tier project. By mid-day, **read quota is exhausted, and writes start failing for everyone**. Plus: They all created test users with names like "Test User", "Admin", "Student", so the database is polluted with duplicate test data that the demo grader will see.

**Why it happens:**  
Spark tier allows 50K reads/day and 20K writes/day. With 5 developers testing in parallel, this is consumed in hours. Plus, without data namespacing, test data from all 5 developers mixes with real app data.

**How to avoid:**  

**1. Create a test data strategy (Phase 1):**
- **Dev environment:** Each developer has a separate branch of test data with a prefix like `test_dev_[initials]_`. Tests use this prefix.
- **CI/staging:** One "golden" copy of test data used by all. Reset before each test run.
- **Production:** Real user data, absolutely no test accounts.

**2. Use Firestore's development mode (24-hour read/write allowance) for active testing:**
```bash
gcloud firestore databases create --region us-central1 --type firestore-native
# This gives you 50K reads/20K writes per day, enough for 5 devs
```

**3. Establish a daily quota check:**
Every morning (or end of day), the "dev lead" checks Firestore Usage in the console:
- If reads > 40K by noon, reduce testing (use Android emulator only, no device testing)
- If writes > 15K by noon, pause integration tests

**4. Add test data helper to the codebase:**
```dart
class FirestoreTestHelper {
  static Future<void> seedTestData(String devInitials) async {
    final prefix = 'test_${devInitials}_';
    
    await FirebaseFirestore.instance
        .collection('users')
        .doc('${prefix}teacher_1')
        .set({
          'name': '${prefix} Teacher',
          'email': '${prefix}teacher@test.local',
          'role': 'teacher',
        });
    // ... more test data
  }
}
```

**Phase 1:** Establish the quota strategy and test data naming convention before anyone writes to Firestore.

**Warning signs:**  
- Firestore reads exceed 30K by 2 PM on testing days
- Database contains dozens of "Test User" accounts from different developers
- Developers can't distinguish real app data from test data
- Quota exhaustion blocks unrelated developers

**Phase to address:** Phase 1 (Foundation) — before Firestore is used for actual development.

### Pitfall 12: Demo Fails on Grader's Machine (Environment Mismatch)

**What goes wrong:**  
The app runs flawlessly on all 5 developers' machines. Demo day arrives. The grader checks out the code, runs `flutter pub get`, builds on Windows, and the app crashes with:
- Firebase not initialized (wrong Flutter SDK version)
- sqflite3.dll not found (Windows SDK version)
- APK too large (release build optimization skipped)
- Splash screen hangs (emulator startup issue)

The team cannot demo the app because it won't build/run in the grader's environment.

**Why it happens:**  
Developers test on their own machines with their own Flutter SDK version, Android SDK version, and Windows SDK setup. The grader uses a standardized lab environment that may differ:
- Different Flutter SDK version (e.g., beta vs. stable)
- Missing Windows SDK or outdated version
- Different CPU architecture (Apple Silicon vs. Intel)
- No Android device/emulator configured
- Environment variables not set correctly

**How to avoid:**  

**1. Define and document a minimum environment:**
Create `ENVIRONMENT.md`:
```markdown
# Memocard Development Environment

## Required
- Flutter 3.24.0 (stable channel) or later
  ```bash
  flutter channel stable
  flutter upgrade
  flutter --version  # Should be 3.24.0+
  ```

- Android SDK 33+ (API level 33)
- Windows SDK 19041+ (for Windows desktop builds)

## Recommended
- Android Studio 2024.1+
- Xcode 15.0+ (if developing on macOS)

## Platform-Specific Setup

### Android
```bash
flutter config --enable-android
flutter doctor
# All Android SDK components must be installed
```

### Windows
```bash
flutter config --enable-windows
flutter doctor
# Visual C++ Build Tools 2022 must be installed
# Windows SDK 19041+ required
```

## Verification
```bash
# Build on all platforms to ensure compatibility
flutter build apk --release
flutter build windows --release
```
```

**2. Add CI/CD to verify builds work in isolation (Phase 2):**
Use GitHub Actions (or similar) to test builds on:
- Windows (flutter build windows --release)
- Android (flutter build apk --release)
- Web (flutter build web --release)

If any build fails, the PR is not merged.

**3. Create a demo checklist (2 days before demo):**
```markdown
## Pre-Demo Verification Checklist

- [ ] All developers `git pull` and `flutter pub get`
- [ ] All platforms build without errors
  - [ ] `flutter build apk --release`
  - [ ] `flutter build windows --release`
- [ ] App launches on Android emulator
- [ ] App launches on Windows desktop
- [ ] Firebase initializes correctly on both platforms
- [ ] Database (sqflite + Firestore) is accessible
- [ ] All 5 assigned developers' modules load and function
- [ ] UI matches Stitch design (no broken layouts)
- [ ] No crashes in the primary flow (Auth → Dashboard → Learn → Quiz)
- [ ] Grader can log in with test credentials
- [ ] All security rules are enforced (grader cannot see other user data)
```

**4. Have the grader (or a neutral party) test the build 1 day before demo:**
Don't wait until demo day to discover the app won't build in the grading environment.

**Phase 3 (Code Freeze):** 2 weeks before demo, verify builds on a clean machine (or CI/CD).

**Warning signs:**  
- Build works on some developers' machines, fails on others
- Environment variables or SDK paths are hardcoded in the app
- Grader's machine has a different Flutter SDK version and builds fail
- App runs in emulator but crashes on real device
- APK works on developer's phone, fails on grader's device

**Phase to address:** Phase 1 (Foundation) — document ENVIRONMENT.md. Phase 3 (Code Freeze) — 2 weeks before demo, test on grader's machine.

### Pitfall 13: Individual Contribution Unclear in Git History

**What goes wrong:**  
Demo day. The grader reviews the git history to assess individual contributions. All commits are by "Team Memocard" with cryptic messages like "Update UI" or "Fix bugs". The grader cannot determine who did what. Grades are deducted because evidence of individual work is missing.

Or: Four developers have 100+ commits each, one developer has 5 commits. The grader assumes that developer did little work, even if they did something critical (like Firebase setup).

**Why it happens:**  
Developers prioritize features over documentation. Commits are made quickly with vague messages. All code is merged to `main` without individual branches, so the grader can't see who worked on what module.

**How to avoid:**  

**1. Use feature branches with developer names:**
```bash
git checkout -b feature/auth-login_alice
git checkout -b feature/quiz-grading_bob
git checkout -b feature/classroom-management_carol
```

**2. Require meaningful commit messages:**
```bash
# GOOD: Specific and traceable to one developer
git commit -m "AUTH: Implement Firebase login flow with email verification"
git commit -m "QUIZ: Add auto-grading for multiple-choice questions"

# BAD: Vague and unhelpful
git commit -m "Update UI"
git commit -m "Fix bugs"
```

**3. Add co-author tags for collaborative commits:**
```bash
git commit -m "AUTH: Add Firebase login and registration

This commit implements:
- Firebase Auth initialization on Windows
- Email/password registration flow
- Session persistence via SharedPreferences
- Login error handling and validation

Co-Authored-By: Alice <alice@example.com>
Co-Authored-By: Bob <bob@example.com>"
```

**4. Keep pull requests open for code review before merging:**
Each developer's PR should be reviewed by at least one other developer. The PR description should clearly state:
- What was built
- Which use cases it fulfills
- Testing done

Example PR:
```markdown
## Description
Implement User Profile screen (UC8–10) with edit and delete functionality.

## Use Cases
- UC8: View Profile
- UC9: Edit Profile
- UC10: Change Password

## Modules Changed
- `screens/profile_screen.dart` (UI)
- `lib/data/repositories/user_repository.dart` (data layer)
- `lib/riverpod/providers/user_provider.dart` (state)

## Testing
- [x] Login as teacher, view profile
- [x] Edit name and email, verify Firestore update
- [x] Change password, verify Firebase auth update
- [x] Delete account, verify cleanup

## Assigned To
@carol (Screen design + implementation)
@alice (Firestore write logic)
```

**Phase 1:** Establish branching and commit message conventions in CONTRIBUTING.md.

**Phase 3 (Code Freeze):** 1 week before demo, review git history to ensure every developer has clear, documented commits.

**Warning signs:**  
- Commits are attributed to "Group 6" instead of individuals
- Most commits have vague messages ("Update", "Fix", "Merge branch")
- One developer has vastly more/fewer commits than others
- Grader cannot identify which developer built which feature

**Phase to address:** Phase 1 (Foundation) — establish conventions. Phase 3 (Code Freeze) — audit history 1 week before demo.

### Pitfall 14: Scope Creep & Skipping MVP Verification (Running Out of Time)

**What goes wrong:**  
The team commits to all 60 use cases. By week 2, it's clear they won't finish. But instead of cutting scope, they try to code faster, skip testing, and push untested features. By demo day, the app is broken and barely functional. The grader sees crashes, missing features, and incomplete UI.

**Why it happens:**  
Course projects have hard deadlines. Pressure mounts as the deadline approaches. Developers add just one more feature instead of shipping a solid MVP. Testing is deferred. Code review is skipped. By demo day, the app is a fragile mess.

**How to avoid:**  

**1. Commit to a realistic MVP (Phase 1):**
The SRS has 60 use cases, but the MVP should be ~30. Example MVP:
- Auth (UC1–3: register, login, logout) — 2 screens
- Flashcard CRUD (UC16–19: create, view, edit, delete) — 4 screens
- Learning Mode (UC24–27: select, learn, save progress, view results) — 3 screens
- Quiz (UC50–55: create, publish, take, grade, view results) — 5 screens

**Total: ~14 screens. Doable in 4 weeks with 5 developers.**

Advanced features (Admin UC5–15, Classroom UC36–49) are Phase 2 if time allows.

**2. Establish Definition of Done for each phase:**
```markdown
## Phase 1 MVP — Definition of Done
- [ ] All 14 MVP screens implemented (match Stitch design)
- [ ] Firestore and SQLite schema complete and tested
- [ ] Firebase Auth flow works (register, login, logout)
- [ ] Flashcard CRUD (create, read, update, delete) fully functional
- [ ] Learning mode: select set → learn → save progress → view results works end-to-end
- [ ] No crashes or unhandled exceptions
- [ ] Code review completed for all PRs
- [ ] Security rules implemented and tested
- [ ] Tested on both Android and Windows
- [ ] README with build/run instructions
```

**3. Track progress with burndown chart (if using Agile):**
Every Friday, plot:
- Planned work (use cases or screens)
- Completed work
- Remaining work

If the curve shows remaining work > 2 weeks of effort, cut scope now.

**4. Stop adding features 1 week before demo:**
Code Freeze at week 3 of 4. Final week is:
- Bug fixes only
- Testing
- Polish
- Demo rehearsal

No new features, period.

**Phase 1:** Agree on MVP scope in week 1. Get sign-off from all 5 developers.

**Warning signs:**  
- By week 2, developers are still debating which features to include
- No clear MVP defined; team is trying to do everything
- By week 3, the team is "almost done" but fundamental features are broken
- Code review is skipped to "save time"

**Phase to address:** Phase 1 (Foundation) — establish MVP scope before coding starts.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hardcode Firebase project ID | Fast setup, no config file | Breaks when switching projects (e.g., dev → staging) | Never — use `firebase_options.dart` from day 1 |
| Skip security rules until demo | Features ship faster | Database is open, data leaks, grader fails you | Never — write rules before data exists |
| Commit generated files (*.g.dart) | Avoid regeneration step | Merge conflicts, out-of-sync generated code | Never — gitignore and regenerate post-pull |
| Mutable state in providers (List.add) | Faster to code | Provider rebuilds silently, UI out of sync | Never — use immutable data structures with `@freezed` |
| Direct Firestore queries in UI | Quick prototyping | N+1 reads, quota exhaustion, no testability | Week 1 only; move to repository layer by week 2 |
| No error handling in async calls | Finish faster | App crashes on network error | Never — handle errors explicitly in all data operations |
| Test on Android only, skip Windows | Save time on setup | Demo fails on grader's Windows machine | Never — test on both platforms weekly |
| One developer owns Firebase config | One source of truth | Single point of failure; knowledge silos | Only until Phase 1 complete; then document for handoff |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| **sqflite_common_ffi + Windows** | Forget `sqfliteFfiInit()` before `databaseFactory = databaseFactoryFfi` | Call `sqlfiteFfiInit()` first, always, in `main()` for Windows/Linux |
| **Firebase + Windows** | Missing `flutterfire configure --platforms=windows` | Run `flutterfire configure` with Windows enabled, verify options in `firebase_options.dart` |
| **Riverpod + Firestore** | Use `.watch()` for one-time reads; use `.read()` for reactive data | `.watch()` for reactive streams, `.read()` for one-time values in callbacks |
| **Firestore + offline cache** | No cache invalidation after sync | Always update local DB `is_synced` flag after Firestore write succeeds |
| **Riverpod + StreamProvider** | Forget `.autoDispose` on Firestore listeners | Always use `.autoDispose` or `ref.onDispose()` to cancel Firestore subscriptions |
| **Firestore + compound queries** | Forget to build composite indexes | Export indexes.json and deploy with `firebase deploy --only firestore:indexes` |
| **Multiple developers + pubspec.yaml** | Concurrent edits cause conflicts | Designate one person per day to add dependencies; merge `pubspec.lock` with `union` strategy |
| **Generated files (.g.dart)** | Commit generated files to git | Gitignore all generated files; regenerate post-pull with `flutter pub run build_runner build` |
| **Firestore + 5 developers** | Shared project, read quota exhausted by noon | Enforce test data prefixes; check quota daily; use separate Firestore instances for heavy testing |
| **Flutter Windows desktop build** | Missing Visual C++ Build Tools or Windows SDK | Run `flutter doctor -v`; install VC++ 2022; set `WINDOWS_SDK_VERSION` environment variable |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| **N+1 Firestore reads** | Firestore quota exhausted; app reads 40K+/day | Denormalize small sets into parent doc; use batch fetch with document references | 5+ flashcards per set × 10 sets × 5 users = 50+ reads for one action |
| **Firestore listener leaks** | Memory usage grows; app sluggish after navigation | Use `.autoDispose` on StreamProvider; add `ref.onDispose()` cleanup | 10+ screens visited in a session = 10+ active listeners = noticeable lag |
| **Unoptimized SQLite queries** | Offline mode is slow; UI freezes on first launch | Add indexes to frequently queried columns (user_id, set_id, synced); use pagination for large result sets | >1000 flashcards in local DB; first load takes >2 seconds |
| **Riverpod provider rebuild storms** | UI rebuild loops; constant refetches | Avoid watching providers in event handlers; use `ref.read()` in callbacks; prevent rebuilds from writes | Changing state inside a watched provider causes infinite rebuild |
| **Image/asset loading without lazy loading** | App startup takes >5 seconds | Lazy-load images; pre-cache critical images on app start; use thumbnails for lists | 50+ flashcard images loaded synchronously |
| **Overfetching data from Firestore** | Unnecessary network calls; high data usage | Only fetch fields needed for current screen; use `.select()` or projection if available | Fetching entire user profile (including settings, preferences) just to display name |

## Security Mistakes

| Mistake | Risk | Prevention |
|---------|------|------------|
| Firestore rules allow unauthenticated reads | Anyone can read all user data, quiz results, classroom info | Require `request.auth != null` for all reads; test in Rules Playground before deploying |
| User ID is guessable (sequential integers) | Attacker can enumerate users and their data by guessing IDs | Use `uuid` for user IDs; never rely on sequential IDs for privacy |
| Sensitive data stored in SharedPreferences | SharedPreferences is unencrypted; anyone with device access can read tokens | Store only non-sensitive data (theme, language); use Flutter Secure Storage for auth tokens |
| Firebase API key exposed in web/desktop config | Attacker can use key to query Firestore directly (if rules are weak) | Write strict security rules; API key exposure is acceptable if rules are locked down |
| Test account credentials hardcoded in app | If test credentials are in source code, attacker has valid login | Use environment variables or config files; never commit credentials |
| No rate limiting on auth endpoints | Brute force attacks on login possible | Firebase Auth handles this; don't implement custom auth bypassing Firebase |
| Firestore rules reference wrong collection | Teacher can accidentally read student data; student can modify quiz answers | Carefully test all path patterns in Rules Playground; require code review for rules changes |

## UX Pitfalls

| Pitfall | User Impact | Better Approach |
|---------|-------------|-----------------|
| No offline indicator | User doesn't know if data is fresh or stale; trusts wrong data | Show "Cached data" or "Offline mode" banner when no network connection |
| Loading spinner for every Firestore fetch | App feels sluggish; waiting for data on every navigation | Use cached data while fetching; show skeleton screens or previous data + refresh button |
| Data disappears when user navigates away | User thinks they lost their data; frustration | Use `.keepAlive(true)` in Riverpod; persist data to SQLite so it's available offline |
| Sync errors silently fail | User doesn't know offline edits didn't sync; data divergence | Show "Pending changes" indicator; display sync errors prominently; allow manual retry |
| Quiz auto-grades incorrectly but no feedback | Student doesn't understand why they got an answer wrong | Show the correct answer + explanation; allow teacher to override grade |
| No evidence of individual contribution visible to user | (Course project) Grader sees commits but can't identify which developer built what | Feature headers/screens with developer name in demo (e.g., "Auth by Alice, Quiz by Bob"); clear git history |
| Demo crashes mid-presentation | Grader sees broken app; grades deducted | Test demo scenario on grader's machine 1 day before; have fallback screenshots/video |

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| **sqflite_ffi not initialized** | LOW | Add `sqlfiteFfiInit()` to main.dart; rebuild Windows app; test database access |
| **Firebase Windows startup crash** | LOW-MEDIUM | Run `flutterfire configure --platforms=windows`; verify Windows options in `firebase_options.dart`; rebuild |
| **Riverpod listener leak** | MEDIUM | Audit all StreamProviders; add `.autoDispose` or `ref.onDispose()` cleanup; rebuild and profile memory |
| **Firestore quota exhausted** | MEDIUM | Implement N+1 mitigation (denormalization, batch fetch); wait for quota reset at midnight Pacific; deploy optimized version |
| **Composite index missing at demo** | MEDIUM | Build index in Firebase console (3–5 min wait); export indexes.json; deploy to test project; rebuild app |
| **pubspec.yaml merge conflict** | LOW | Resolve conflict manually; run `flutter pub get`; verify no dependencies are missing; test build |
| **Generated files out of sync** | LOW | Run `flutter pub run build_runner build`; verify generated files match source; commit if needed (or re-gitignore) |
| **Security rules too permissive** | HIGH | Write proper rules; test in Rules Playground; deploy to Firestore; audit existing data for leaks |
| **5 devs exhausted Spark quota** | HIGH | Implement test data prefixes; reset database; set quota alerts; request budget increase or switch to Blaze tier |
| **Demo fails on grader's machine** | HIGH | Debug environment differences (Flutter SDK, Android SDK, Windows SDK); document ENVIRONMENT.md; test on clean machine |
| **Individual contributions unclear** | MEDIUM | Rewrite git history with meaningful messages + developer tags (use `git commit --amend --author` carefully or document commits retroactively in README) |
| **Scope creep, running out of time** | HIGH | Cut non-MVP features immediately; focus on completing + testing MVP; skip polish; accept demo with fewer features than intended |
